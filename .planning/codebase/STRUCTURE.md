## Directory Purposes

**src/:**
- Purpose: All application source code
- Contains: Modular JavaScript files, each with single responsibility
- Key files: `index.js` (orchestrator), `*.js` (feature modules), `__tests__/` (test suite)

**src/__tests__/:**
- Purpose: Unit and integration tests using Node.js native test runner
- Contains: One test file per source module (pairs with parallel src/ file)
- Key files: All `.test.js` files; index.test.js is integration-level

**dist/:**
- Purpose: Bundled artifact for GitHub Actions runtime
- Contains: Single `index.js` file produced by @vercel/ncc build
- Generated: Yes (built via `npm run build`)
- Committed: Yes (required because GitHub Actions runs from dist/)

**.github/workflows/:**
- Purpose: CI/CD automation
- Contains: `ci.yml` workflow for test + build on push
- Key workflow: Runs tests, builds dist/, optionally publishes

**.planning/codebase/:**
- Purpose: GSD (guided structured documentation) analysis outputs
- Contains: ARCHITECTURE.md, STRUCTURE.md, etc.
- Generated: Yes (by GSD mapper)
- Committed: Yes (for reference)

**docs/plans/:**
- Purpose: Implementation plans generated by GSD planner
- Contains: Markdown phase plans with task breakdowns
- Generated: Yes (by GSD planner)
- Committed: Yes (for audit trail)

## Key File Locations

**Entry Points:**
- `src/index.js`: Main action entry point (exported `run()` function, auto-runs if DISABLE_AUTORUN !== '1')
- `dist/index.js`: Bundled artifact executed by GitHub Actions (produced from src/index.js)
- `action.yml`: GitHub Actions metadata (defines inputs/outputs, specifies main: 'dist/index.js')

**Configuration:**
- `action.yml`: Action definition (inputs, outputs, branding, runtime)
- `package.json`: Node.js project metadata, dependencies, build scripts
- `.github/workflows/ci.yml`: CI workflow definition

**Core Logic:**
- `src/contributions.js`: GraphQL query for contributions + calendar heatmap
- `src/activity.js`: REST API events fetcher with pagination and deduplication
- `src/repos.js`: REST API repositories fetcher with filtering (non-fork, non-archived)
- `src/streak.js`: Calculates current/longest streak from calendar
- `src/stats.js`: Calculates weekly/monthly statistics from calendar and activity

**Utilities:**
- `src/inputs.js`: Positive integer parser with strict validation
- `src/schema.js`: Output structure validator (type assertions)
- `src/lang-colours.js`: Language-to-hex-color lookup table

**Testing:**
- `src/__tests__/*.test.js`: Test suite (one test file per source module)
- Test helpers: Mocks for Octokit, core API, event factories

## Naming Conventions

**Files:**
- `<module>.js`: Source files use camelCase (e.g., `contributions.js`, `lang-colours.js`)
- `<module>.test.js`: Test files mirror source name with `.test.js` suffix
- `index.js`: Conventional entry point name
- `action.yml`: GitHub Actions standard name

**Directories:**
- `src/`: Standard location for application code
- `src/__tests__/`: Node.js test convention (discoverable by test runners)
- `dist/`: Conventional build output directory
- `.github/`: GitHub Actions and workflows
- `docs/`: Documentation and planning

**Functions:**
- `camelCase` for all functions (e.g., `fetchContributions`, `calculateStreak`, `parsePositiveInt`)
- Verb + noun pattern (fetch*, calculate*, parse*, map*, validate*)
- Descriptive names (no abbreviations except established conventions like max, iso)

**Variables:**
- `camelCase` for variables and parameters
- `UPPER_CASE` for constants (e.g., `CONTRIBUTION_LEVEL_MAP`, `EMPTY_CONTRIBUTIONS`)
- Descriptive names (topActivities, seenUrls, contributionCalendar)

**Types (no TypeScript, but pattern implicit in JSDoc):**
- Activity items: `{ type, repo, repoUrl, title, url, date, meta? }`
- Calendar: `{ weeks: [{ days: [{ date, count, level }] }] }`
- Contributions: `{ total, commits, pullRequests, pullRequestReviews, issues, restricted }`

## Where to Add New Code

**New Feature (e.g., add watchers count to repos):**
- Fetcher modification: `src/repos.js` (update mapRepo, fetch additional field)
- Schema update: `src/schema.js` (add validation for new field)
- Test update: `src/__tests__/repos.test.js` (add assertion for new field)
- Example: Adding `watchers` to repository object requires changes in all three files

**New Data Source (e.g., gists):**
- New fetcher file: `src/gists.js` (follow pattern of contributions/activity/repos)
- Entry point: `src/index.js` (add fetch via Promise.allSettled, fallback shape, warnings)
- Schema: `src/schema.js` (add validation for new top-level key)
- Test file: `src/__tests__/gists.test.js` (unit and integration tests)
- Calculation: `src/stats.js` if new stats are needed
- Output top-level: Add to data object in index.js and schema

**New Calculation (e.g., language distribution):**
- New calculation file: `src/languageStats.js`
- Call in: `src/index.js` after all fetchers resolve
- Schema: `src/schema.js` (add validation for new stats shape)
- Test file: `src/__tests__/languageStats.test.js`
- Dependencies: Pass calculated data (likely `repositories`) as parameter

**Utilities (e.g., date formatter):**
- Shared helpers: `src/utils.js` (if generic) or within specific module
- If truly shared, create new file or extend existing utility
- Test: Add corresponding tests in `src/__tests__/<util>.test.js`
- Example: `toMillis` in activity.js is activity-specific; date utility would go to utils.js

**Environment-Specific Code:**
- GitHub Actions integration: Keep in `src/index.js` (core.info, core.warning, core.setFailed)
- Pure logic: Keep separate from @actions/core dependency (easier to test)
- Dependency injection: Allow overrides in test mode (already pattern in index.js)

## Special Directories

**dist/:**
- Purpose: Bundled GitHub Actions artifact
- Generated: Yes, by `npm run build` (via @vercel/ncc)
- Committed: Yes (GitHub Actions runs this, not src/)
- Update: Never edit manually; always rebuild via npm run build

**.github/workflows/:**
- Purpose: CI/CD automation
- Generated: No (manually created and maintained)
- Committed: Yes
- Update: Via .github/workflows/*.yml files

**.planning/codebase/:**
- Purpose: GSD analysis outputs for future reference
- Generated: Yes (by GSD mapper)
- Committed: Yes (for audit trail)
- Update: Via /gsd:map-codebase command

**node_modules/:**
- Purpose: Installed dependencies
- Generated: Yes (from package-lock.json)
- Committed: No (.gitignored)
- Update: Via npm ci or npm install

## Build & Output Process

**Development:**
```bash
npm ci              # Install dependencies (uses package-lock.json)
npm test            # Run tests via node --test
npm run build       # Bundle src/index.js → dist/index.js with @vercel/ncc
```

**GitHub Actions Execution:**
1. Workflow runs job on runner (ubuntu-latest)
2. GitHub Actions runtime loads `dist/index.js` (specified in action.yml)
3. Action runs with injected inputs from action.yml/workflow
4. Outputs written to GitHub Actions context
5. Bundled artifact means no node_modules/ needed at runtime

**Bundling Details:**
- Tool: @vercel/ncc (Next.js compiler)
- Input: `src/index.js`
- Output: `dist/index.js` (single file with all dependencies inlined)
- Advantage: No external dependencies required at runtime

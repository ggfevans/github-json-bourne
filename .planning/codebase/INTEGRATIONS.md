# External Integrations

**Analysis Date:** 2026-02-21

## APIs & External Services

**GitHub REST API:**
- Service: GitHub REST API v2022-11-28
- What it's used for: Fetching user repositories and public events
- SDK/Client: `@octokit/rest` (v21.1.1)
- Auth: GitHub token via `auth` parameter or `GITHUB_TOKEN`/`GH_TOKEN` environment variables
- Endpoints used:
  - `GET /users/{username}/repos` - Fetch user repositories (paginated, 100 per page)
  - `GET /repos/{owner}/{repo}/compare/{basehead}` - Compare commits for detailed push activity

**GitHub GraphQL API:**
- Service: GitHub GraphQL endpoint
- What it's used for: Fetching contribution statistics and contribution calendar
- SDK/Client: `@octokit/graphql` (v8.2.1)
- Auth: GitHub token via `bearer {token}` authorization header
- Query: Custom GraphQL query fetches:
  - Total contribution counts (commits, PRs, PR reviews, issues)
  - Restricted contributions count
  - Contribution calendar with daily heatmap data

## Data Storage

**Output Format:**
- JSON file written to local filesystem (default: `github.json`)
- Configurable path via `output-path` input parameter
- Data includes:
  - Contribution statistics (total, commits, PRs, reviews, issues, restricted)
  - Contribution calendar (weeks with daily contribution levels 0-4)
  - Recent activity (commits, PRs, issues up to max-activities limit)
  - Repository data (name, description, language, stars, URL)
  - Streak calculation
  - Stats calculation

**Persistence:**
- File-based storage in repository or workflow artifact
- No database integration

## Authentication & Identity

**Auth Provider:**
- GitHub-native authentication
- Accepts personal access tokens (PAT) or GitHub Actions workflow token (`GITHUB_TOKEN`)
- Token scopes determine data access:
  - Public scope: Public contributions only
  - Full scope: Includes private contributions if PAT has access

**Implementation:**
- Token passed via `token` input parameter
- Falls back to environment variables `GITHUB_TOKEN` or `GH_TOKEN`
- Authentication headers injected into all API requests

## Monitoring & Observability

**Error Tracking:**
- Not integrated - errors logged via `@actions/core` warning/info methods
- Best-effort fallbacks on error (returns empty data for failed endpoints)

**Logs:**
- GitHub Actions logging via `@actions/core`
  - `core.info()` - Information messages (e.g., "Fetching GitHub data for {username}")
  - `core.warning()` - Non-fatal errors (e.g., pagination failures, fallback usage)
  - `core.setFailed()` - Fatal errors that cause action failure

## CI/CD & Deployment

**Hosting:**
- GitHub Actions - Runs as containerised action on GitHub runners

**Action Distribution:**
- Bundled JavaScript file (`dist/index.js`) generated by ncc
- Published to GitHub Actions marketplace via repository
- Inputs configurable by workflow users

**Workflow Inputs:**
- `username` - GitHub user to fetch data for (defaults to `github.repository_owner`)
- `token` - GitHub token for API access (defaults to `github.token`)
- `output-path` - Path for JSON output (default: `github.json`)
- `max-repos` - Maximum repositories to include (default: 12)
- `max-activities` - Maximum activities to include (default: 30)
- `max-pages` - Maximum event pages to scan (default: 3)

## Environment Configuration

**Required env vars:**
- `GITHUB_TOKEN` or `GH_TOKEN` - GitHub API authentication
- `GITHUB_REPOSITORY_OWNER` - Used as default username if not provided

**Optional env vars:**
- `GITHUB_JSON_BOURNE_DISABLE_AUTORUN` - Set to '1' to skip auto-execution (used in tests)

**Secrets location:**
- GitHub Actions Secrets (accessed as `github.token` or via PAT in CI context)

## Pagination & Rate Limiting

**REST API Pagination:**
- Default: 100 items per page
- User repos: Fetches until `maxRepos` reached or no more repos available
- User events: Fetches up to `maxPages` (default 3) with 100 events per page

**GraphQL Limitations:**
- Single query per request (no pagination within GraphQL response)
- Returns all calendar weeks in one request

**Rate Limits:**
- GitHub API has standard rate limits (60 req/hour unauthenticated, 5000 req/hour authenticated)
- No explicit rate limit handling or backoff strategy in code
- Pagination stops gracefully on error after first page (falls back to partial data)

## Webhook & Callbacks

**Incoming:**
- None - Action is invoked by GitHub workflow events, not webhooks

**Outgoing:**
- None - Action outputs JSON file and action outputs only, no external callbacks

## API Error Handling

**REST API:**
- First page failure throws error (action fails)
- Subsequent page failures logged as warning, pagination stops, returns partial results

**GraphQL API:**
- Failure throws error with message context (action fails)
- No fallback - contributions data critical for output validation

**Repo Fetch Errors:**
- Paginated fetch: First page error throws, subsequent pages fallback to warning
- Filters out forks and archived repositories

**Event Fetch:**
- First page required (throws on failure)
- Subsequent pages graceful failure with warning and break
- Fallback for large push events: Attempts to fetch detailed commits via compare endpoint, falls back to push payload commits
